# 世界杯 AI 预测分析工具 — PRD v0.1

> 版本：2026-06-01 v0.1
> 范围：只覆盖产品逻辑与功能（输入 / 输出 / 流程）；定价 / 商业模式暂不讨论

---

## 一、产品定位

**一句话**：用户输入一个问题或粘贴一个 Polymarket 链接，AI 直接返回多 Agent 共识的预测分析。

**不做什么**（同等重要）：
- ❌ 不做赔率比对工具（OddsFlow 在做）
- ❌ 不做每日 tips 推送（NerdyTips 在做）
- ❌ 不做交易执行 / 下单（用户自己去 Polymarket 下）
- ❌ 不做社区 / 跟单 / 排行榜
- ❌ 不做新闻聚合站

**只做一件事**：**对一个具体的 Polymarket 市场或用户问题，给出 AI 多 Agent 共识的预测分析。**

---

## 二、核心用户场景

> "我看到 Polymarket 上有一个市场'巴西能否进决赛'当前价 38%，我想问 AI 这个价合不合理。"

1. 用户打开产品 → 看到一个输入框
2. 粘贴 Polymarket 链接 / 或直接打字问问题
3. 等待 ~10-30 秒，AI 多 Agent 在后台并行分析
4. 看到结构化分析结果（置信度 + 推理 + 关键数据）
5. 满意 → 点"去 Polymarket 下注"跳转 / 不满意 → 追问

整个流程**没有登录、没有付费墙、没有引导**——和 ChatGPT 一样直接用。

## 三、输入设计

### 3.1 输入框（产品唯一的核心入口）

一个单一输入框，支持三种输入：

| 输入类型 | 识别方式 | 处理逻辑 |
|---------|---------|---------|
| **Polymarket 市场链接** | 正则匹配 `polymarket.com/event/...` 或 `polymarket.com/market/...` | 自动抓取该市场元数据（标题、当前价、解算标准、订单簿）后进入分析 |
| **自然语言问题（已挂市场）** | LLM 语义匹配到现有 Polymarket 市场 | 同上，并在结果里展示"我们认为你问的是这个市场" |
| **自然语言问题（无对应市场）** | 找不到匹配市场 | 直接基于问题给出 AI 分析（无市场价对比） |

**辅助元素**：

- 占位文字示例（每次刷新轮换）：
  - "粘贴 Polymarket 链接"
  - "巴西能进决赛吗？"
  - "梅西世界杯能进 3 球以上吗？"
  - "Will France win Group D?"
- 输入框下方**3-4 个示例链接**（一键填入）：当前热门世界杯市场

### 3.2 输入的附加选项（可选，默认收起）

折叠区，给进阶用户：

- **追加上下文**：粘贴一段直播信息 / 推特帖 / 新闻链接，让 AI 把这条信息纳入分析
- **指定 Agent 视角**：默认全部 4 个 Agent；进阶用户可以只挑某个 Agent（数据派 / 盘口派 / 新闻派 / 战术派）
- **语言**：中 / 英（默认跟浏览器）

### 3.3 首页主体（输入框下方分两个并列板块）

> 多数用户进来时**还没想好要问什么**。首页主体分两块：
>
> - **板块 ①「赛事 Live」**——让用户"看到正在发生的事"
> - **板块 ②「分析工具」**——让用户"找到值得分析的问题"

```

│  Hero：Logo + 一句话定位 + 输入框            │
└─────────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────────┐
│  板块 ①  赛事 Live                            │
│  ├─ (a) 赛事动态数据                         │

│  └─ (c) 预告模块（即将开赛）                  │
└─────────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────────┐
│  板块 ②  分析工具                             │
│  ├─ 🔥 Hot Markets（长期热门市场）            │
```

## 板块 ②「分析工具」

> **回答用户的问题**：除了眼前的比赛，还有什么值得分析？大家都在问什么？

这个板块的内容**与具体的某一场比赛解耦**，聚焦"跨比赛、跨日期"的热门内容。

**卡片类型**：

### (b) Suggested Questions — 智能问题清单

最下方一栏 "用户也在问"，6-10 条文字 chip。

```
💡 用户也在问

[ 内马尔今晚会进球吗？ ]    [ 美国队能小组出线吗？ ]
[ 决赛会下雨吗？ ]          [ 阿根廷夺冠概率合理吗？ ]
[ 阵容公布会怎么影响赔率？ ]  [ Polymarket vs Bet365 哪个更准？ ]
```

**生成逻辑**（按优先级）：

1. **热点驱动**：最近 1 小时高频被问的问题
2. **赛事节奏驱动**：开赛前 2 小时 → 阵容类；比赛中 → 在场情况类；结束后 → 复盘 + 下一场预热
3. **行情异动驱动**：某市场 1 小时内变动 > 5% → "为什么 X 的赔率突然变了？"
4. **新闻事件驱动**：突发新闻（伤情、停训）→ "X 受伤会影响夺冠概率多少？"

点击任意 chip = 直接以该问题为输入发起分析。

### 3.4 设计原则

1. **板块 ① 是"看戏"，板块 ② 是"提问"**：
   - 板块 ① 信息密集（实时数据 + 多市场赔率），用户**消费内容**
   - 板块 ② 内容稀疏（卡片 + 问题），用户**找入口**
2. **板块 ① 必须比 Polymarket 自己的赛事页更聚合**——同一比赛对应的多市场赔率聚合到一张面板里是核心竞争力
3. **不堆数据**：板块 ① 的 LIVE 卡片只显示用户一眼能消化的数据；专业指标（让球、BTTS 详细分布）都进结果页 Block 4
4. **巨鲸动向**是差异点：Polymarket 自己的页面不突出，我们要做成醒目元素
5. **实时性**：
   - 比分 / 关键事件：实时
   - 场面数据（控球/射门/xG）：60s 刷新
   - Polymarket 概率：10s 刷新
   - 天气：30min 刷新

## 四、输出设计（产品的真正主体）

输出是一张**结构化的分析报告卡片**，自上而下分 6 个区块：

### Block 1 — 市场快照（最顶部，2 秒能看完）

```
┌─────────────────────────────────────────┐
│  巴西能否进决赛？                          │
│  Polymarket · World Cup 2026             │
│                                          │
│  当前市场价      AI 共识价     Edge       │
│    38%           52%        +14 个点      │
│                                          │
│  [去 Polymarket 看 →]                    │
└─────────────────────────────────────────┘
```

**包含**：市场标题、平台标签、**当前市场价 vs AI 共识价 vs Edge**、跳转链接。

**核心信号**：用颜色（绿 / 红）一眼标出 AI 与市场的差异方向。

### Block 2 — 多 Agent 共识可视化（差异化卖点）

四个 Agent 各自独立给出概率估计，可视化"是否一致"。

```
                   Agent 估计      置信度    一致性
📊 数据派 Agent      54%             高       ✓
💰 盘口派 Agent      48%             中       ✓
📰 新闻派 Agent      55%             高       ✓
⚽ 战术派 Agent      51%             中       ✓
─────────────────────────────────────
   AI 共识           52%             高       4/4
```

**关键设计**：
- 4 个 Agent 一致时，给一个明显的"高置信度"标签
- 出现分歧时，**显式高亮**那个不一致的 Agent，并展开它的理由
- 共识价不是简单平均，是按各 Agent 置信度加权

### Block 3 — 推理摘要（用户停留最久的区块）

一段 200-400 字的自然语言总结，三段式：

> **第一段：市场怎么看。** Polymarket 当前定价 38%，过去 24 小时下跌 5 个点，主要因为内马尔伤情的传闻。
>
> **第二段：AI 怎么看。** 综合 4 个 Agent，我们认为合理价 52%。数据派看球队近期 xG 与对手强度匹配支持更高概率；盘口派看 Bet365 1.85 隐含 54%，与 Polymarket 有显著背离；新闻派确认内马尔已出院，传闻夸大；战术派看 4-2-3-1 阵型对四强对手胜率历史 58%。
>
> **第三段：风险因子。** 如果半决赛抽签遇到法国或西班牙，AI 估计降至 41%。如果阵容公布前 24 小时有变化，建议重新询问。

### Block 4 — 数据展开（折叠，默认收起）

按数据类别折叠展开，给愿意深入的用户：

- **市场数据**：当前订单簿、24h 价格走势图、巨鲸成交
- **球队基础数据**：最近 5 场结果、xG、对手强度、FIFA/ELO
- **即时变量**：阵容、伤病、新闻摘要（带原文链接）
- **跨盘口对比**：Bet365 / Pinnacle / DraftKings 隐含概率 vs Polymarket

### Block 5 — 关键问题（追问入口）

3-5 个**AI 自动生成的相关问题**，点击即可作为新输入：

- "如果内马尔最终伤退，估计会变多少？"
- "和西班牙进决赛的概率呢？"
- "Polymarket 同期 Vinicius 进球榜冠军价合不合理？"

### Block 6 — 一键跳转

- 主 CTA：**[去 Polymarket 下注 →]**
- 次 CTA：**[查看历史预测准确率]**（信任锚）
- 三 CTA：**[分享这份分析]**（生成卡片图，对应社媒传播）

## 五、多 Agent 架构（功能层）

> 不涉及实现细节，只说每个 Agent **看什么数据、输出什么**

| Agent | 输入数据 | 输出 |
|------|---------|------|
| **📊 数据派 Agent** | StatsBomb / FBref 历史比赛、xG、H2H、FIFA/ELO | 基于球队真实实力的概率估计 |
| **💰 盘口派 Agent** | Polymarket CLOB、Bet365/Pinnacle 实时赔率、巨鲸成交 | 基于市场共识 + sharp action 的概率估计 |
| **📰 新闻派 Agent** | NewsAPI / Twitter 球队官号 / 阵容伤病新闻 | 基于即时信息的概率调整 |
| **⚽ 战术派 Agent** | 战术阵型、教练偏好、关键球员表现 | 基于比赛结构的概率估计 |

**协调层**：
- 4 个 Agent 并行执行
- 每个 Agent 输出"概率 + 置信度 + 1 段理由"
- 协调器按置信度加权得到共识价
- 如果分歧大（标准差 > 阈值），显式标出分歧并解释

## 七、关键交互流程

### 7.1 主流程时序

```
用户        前端           编排层          各 Agent           数据源
 │           │              │                │                │
 │─输入─────▶│              │                │                │
 │           │─分析请求─────▶│                │                │
 │           │              │─识别市场─────────────────────────▶│
 │           │              │                │                │
 │           │◀─骨架屏──────│                │                │
 │           │              │─并行调度─────▶ 4 Agents          │
 │           │              │                │─各自拉数据──────▶│
 │           │              │                │◀─返回估计────────│
 │           │              │◀─4 个结果─────│                │
 │           │              │─加权 + 生成报告 │                │
 │           │◀─完整报告────│                │                │
 │◀─渲染────│              │                │                │
```

**响应时间预算**：
- 0-1 秒：骨架屏显示
- 1-5 秒：市场快照（Block 1）已出
- 5-15 秒：多 Agent 估计（Block 2）已出
- 15-30 秒：完整报告渲染完成

**逐块流式渲染**——不让用户干等 30 秒看白屏。

### 7.2 追问流程

用户在结果页底部继续输入 → 保留上一次的上下文（市场 ID + 上次分析结果）→ AI 增量分析 → 在原页面追加新 Block。

### 7.3 异常处理

| 场景 | 行为 |
|------|------|
| Polymarket 链接已解算 | 显示真实结果 + AI 当时是否准 |
| Polymarket 链接流动性极低 | 提示"市场太薄，AI 分析仅作参考" |
| 自然语言匹配不到市场 | 显示"未找到对应市场，是否要 AI 直接回答？" |
| 某个 Agent 数据源失败 | 显示"3/4 Agent 共识"，不让一个 Agent 故障阻塞全流程 |
| 比赛已开始 | 显示"比赛进行中，分析为赛前数据" + 比分 + 最近重新分析时间 |

## 十、设计原则

1. **像 ChatGPT 一样简单**：一个输入框、一份输出，没有引导、没有教程
2. **流式输出，不让用户干等**：Block 1 优先、Block 2 次之、其余渐进出现
3. **可解释性 > 准确率宣传**：宁可说"4 Agent 中 1 个不同意"，也不说"准确率 75%"
4. **信任锚天天可查**：历史准确率页一定要从 day 1 上线
5. **跳转友好**：Polymarket 链接是结果页一等公民，不要为留住用户而阻断流程
